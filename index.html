    Engineering Philosophy

    How I approach scalability, structure, and production.

    Scalability

    Design for growth: modular layers, clear boundaries, testable code. Mobile → API → DB with clear contracts.

    Debugging & Production

    Structured logging, crash reporting, metrics. Reproduce, isolate, fix. No guesswork in production.

    Codebase Structure

    Clean Architecture: domain, data, presentation. MVVM on UI. Dependency injection for testability and clarity.

    My Coding Journey

    Covid gave me something unexpected: time. I’d always been curious about building things on a screen. So I started. At first it was messy — a lot of googling, a lot of “why does this crash?” But I fixed my first real bug in a real app, and something clicked.

    I soon realised that writing code alone isn’t enough. You have to understand the framework you’re using and the architecture behind the code — how layers fit together, where logic lives, and why it’s structured that way. That mindset changed how I built everything after.

    I built apps from scratch, learned to structure code so it didn’t turn into a knot, and learned to work with others — not just my own files, but a shared codebase and a team. Shipping on time and writing code others could read became the goal.

    Over the years I touched more and more: multiple apps, talking to servers, storing data, working closely with design and backend. Code reviews and “how do we keep this maintainable?” became part of the day. I stopped only fixing; I started owning.

    Today I still start with a problem, then design and build. I care about structure, clarity, and shipping. The stack evolves, but the aim doesn’t: build something that works, lasts, and feels right. Still learning. Still shipping.
